# 每日行业简报｜2026-07-23

## 今日核心判断

**企业 AI 正在从“模型能力竞争”进入“企业 Agent 控制平面竞争”。** 最新信号显示，云厂商、数据库厂商、图数据库厂商和企业软件厂商都在围绕同一件事重构产品：让 Agent 能够在企业内部安全地发现数据、调用工具、执行任务、接受治理，并留下可追溯证据。

这对图数据库、本体和数据治理是一个关键窗口。过去图数据库常被定位为复杂关系查询或 GraphRAG 组件；现在更大的机会是成为企业 Agent 的关系控制层：管理 Agent、用户、数据、模型、工具、规则、业务对象和执行结果之间的关系。未来真正有价值的不是“让模型多查一点数据”，而是“让机器行动在业务语义和治理边界内发生”。

---

## 一、AI 与 Agent：企业软件厂商开始争夺 Agent 落地入口

### 1. OpenAI 正在从模型供应商走向企业软件和交付服务

Business Insider 报道称，OpenAI 推出面向企业的软件产品 Presence，用于把 AI Agent 接入企业内部数据、软件、政策和工作流，覆盖客户支持、销售、账单处理、保险理赔、IT 服务请求等流程。报道中特别值得关注的是，Presence 不只是一个聊天界面，而强调模拟、护栏、人类审核、评估 Agent 表现，以及让 Agent 在限定数据和政策范围内行动。

参考：[OpenAI is launching new corporate software that takes it beyond the AI model war](https://www.businessinsider.com/openai-presence-corporate-software-customer-service-sales-2026-7)。

这说明模型厂商正在意识到：单纯卖 API 会被价格竞争和开源模型挤压，真正的企业价值在于把模型嵌入业务流程，并承担落地后的治理、评估和变更管理。WSJ 同时报道 OpenAI 正在扩大企业服务和前线部署工程能力，通过 DeployCo 和咨询伙伴网络帮助企业把 AI 变成真实生产力。

参考：[OpenAI's High-Stakes, High-Touch Push to Make AI Work for Business](https://www.wsj.com/cio-journal/openais-high-stakes-high-touch-push-to-make-ai-work-for-business-7f08b7d4)。

对图数据库和本体厂商的启发是：客户不会只为“接入大模型”付费，而会为“让大模型进入真实流程并可控地执行”付费。图数据库如果能描述业务对象、流程、权限和证据链，就有机会进入 Agent 落地的核心架构。

### 2. Agent 治理必须从制度文本变成技术控制

Gartner 近期文章强调，AI 治理不能停留在政策层面，必须变成可执行、可嵌入、可持续的技术控制。TechRadar 也报道，随着企业 Agent 数量快速增加，很多公司仍缺少集中治理方式，容易出现 AI sprawl：系统碎片化、重复建设、权限不一致、审计困难和成本不可控。

参考：[AI Governance Requires More Than Policies](https://gcom.pdo.aws.gartner.com/en/articles/ai-governance-trism)、[As AI scales, is meaningful governance possible?](https://www.techradar.com/pro/as-ai-scales-is-meaningful-governance-possible)。

这直接指向一个产品机会：Agent Control Plane。它至少要管理五类对象：

- Agent：所有者、用途、模型版本、风险等级、生命周期；
- Context：数据来源、本体版本、时间有效性、敏感等级；
- Tool：可执行动作、参数约束、外部依赖、调用成本；
- Policy：权限、审批、地域、时间、客户类型和监管限制；
- Evidence：输入、计划、工具调用、规则命中、输出和业务影响。

这些对象天然是图结构。Agent 治理不是简单的权限表，而是一张动态关系网络。

## 二、图数据库与 GraphRAG：从“知识增强”进入“运行时治理”

### 1. Neo4j 与 TigerGraph 的最新方向说明 GraphRAG 正在工程化

Neo4j 当前数据库最新版本是 2026.06.0，继续围绕 GraphRAG、向量搜索、Connected Intelligence 和企业知识图谱强化 AI 叙事。Neo4j 的重点不只是“图查询”，而是让 AI 的上下文建立在连接型数据之上，从而提升可解释性和可信度。

参考：[Neo4j Database Release Notes](https://neo4j.com/release-notes/database/)。

TigerGraph 方面，TigerGraph Server 4.2.4 LTS 于 2026 年 7 月 20 日发布；4.3.0-rc1 Preview 则在 7 月 8 日发布，重点包含 mTLS、API、安全访问控制、数据流连接器和管理门户增强。TigerGraph 的 GraphRAG 项目也在 2026 年持续更新，加入 Admin UI、角色访问和按图配置 chatbot LLM 等能力。

参考：[TigerGraph 4.2 Release Notes](https://www.tigergraph.com/docs/tigergraph-server/4.2/release-notes/)、[TigerGraph 4.3 Release Notes](https://www.tigergraph.com/docs/tigergraph-server/4.3/release-notes/)、[TigerGraph GraphRAG](https://github.com/tigergraph/graphrag)。

这说明图数据库竞争已经进入两个并行方向：

1. AI 能力：GraphRAG、向量检索、自然语言查询、Agent 工具集成；
2. 生产能力：认证、安全、API、运维、审计、连接器和企业集成。

只强调 GraphRAG 已经不够。客户会问：这个 GraphRAG 是否能被权限控制？是否能审计？是否能进入企业身份体系？是否支持多租户和成本隔离？是否能解释 Agent 使用了哪些路径和证据？

### 2. Agentic GraphRAG 在金融非结构化数据中价值突出

论文《Agentic GraphRAG: Navigating Unstructured Financial Data with Collaborative AI》以商业登记数据为对象，提出协作式 Agentic GraphRAG：先从结构化字段构建强节点，再从非结构化公告中抽取弱节点，随后做身份解析和去重，再由模块化 Agent 完成意图路由、受限反思、图工具访问和状态感知回答。论文强调人机协同仪表盘用于暴露证据和执行轨迹。

参考：[Agentic GraphRAG: Navigating Unstructured Financial Data with Collaborative AI](https://arxiv.org/abs/2605.18770)。

这对金融客户非常有启发。企业工商、司法公告、监管处罚、舆情、财报、招投标和供应链合同都具有“结构化记录 + 非结构化文本 + 时间变化 + 实体歧义”的特点。向量 RAG 容易漏掉多跳关系和实体演化，传统规则抽取又覆盖不足。Agentic GraphRAG 的优势在于把确定性结构、弱抽取、身份解析和图查询结合起来。

可落地场景包括：

- 授信审查：企业、实控人、关联公司、司法风险、舆情和财务指标联动分析；
- 反洗钱：账户、交易、设备、地址、商户、受益人的多跳网络；
- 监管报送：规则、指标、字段、血缘、校验关系和报送口径统一；
- 投研：公告、财报、行业事件、供应链关系和价格波动的证据网络。

## 三、数据库与开源技术：数据库正在成为 Agent 的数据执行层

### 1. PostgreSQL 继续推动图、AI 和工程生态融合

PostgreSQL 19 Beta 2 已于 2026 年 7 月 16 日发布，官方强调这是功能预览和兼容性测试版本，不建议用于生产。PostgreSQL 19 系列的重要意义在于，SQL/PGQ 属性图能力、性能改进和可观测性能力正在进入主流数据库生态。

参考：[PostgreSQL 19 Beta 2 Released](https://www.postgresql.org/about/news/postgresql-19-beta-2-released-3350/)、[PostgreSQL Release Notes](https://www.postgresql.org/docs/release/)。

这对专用图数据库是压力也是机会。压力在于客户会问：“既然 PostgreSQL 也能做属性图查询，为什么还需要图数据库？”机会在于 SQL/PGQ 会扩大图查询的市场认知，让更多客户理解关系网络分析的价值。

图数据库应该主动分层表达：

- PostgreSQL SQL/PGQ 解决的是让关系数据具备图查询入口；
- 专用图数据库解决的是大规模多跳遍历、路径分析、图算法、实时关系计算、分布式图存储、GraphRAG 和 Agent 治理。

### 2. EDB 与 Oracle 说明数据库厂商正在争夺 Agent 入口

EDB 在 EDB Postgres AI 中提出 Agentic Database、融合分析和治理能力，强调 Agent 时代治理必须在数据所在位置执行，依托 Postgres 角色、行级安全和审计机制控制 Agent 对数据的访问。

参考：[EDB Launches Agentic Database, Converged Analytics, and Governance](https://www.enterprisedb.com/press-releases/edb-launches-agentic-database-converged-analytics-and-governance-bringing-sovereign)。

Oracle Autonomous AI Database 则支持通过 A2A 协议暴露 Select AI agent teams，使外部兼容 A2A 的客户端 Agent 能够发现和调用数据库内 Agent，同时保持 Agent 逻辑和数据访问在数据库内部。

参考：[Oracle Autonomous Database Serverless Release Notes](https://docs.oracle.com/en-us/iaas/releasenotes/services/autonomous-database-serverless/index.htm)。

这说明数据库厂商的目标不再只是“被 Agent 查询”，而是成为 Agent 网络中的可信数据执行环境。图数据库需要找到自己的差异化：不必替代每个事务数据库，而应成为跨系统关系、语义、证据和影响分析的全局治理层。

## 四、金融科技、本体和数据治理：AI 治理进入“证据链”时代

### 1. 金融行业的 AI 落地依赖强数据基础

Rathbones Asset Management 与 Snowflake 的数据基础建设案例表明，受监管行业在尝试 agentic technologies 之前，首先要建立可信数据底座、质量规则和统一数据源。报道提到，该机构围绕 performance、risk、sustainability 等领域建设数据集，并测试 Snowflake agentic technologies 进行趋势和投资表现分析。

参考：[How Rathbones Asset Management built strong data foundations](https://www.itpro.com/business/data-and-insights/how-rathbones-asset-management-built-strong-data-foundations)。

这说明金融 AI 落地不是“先上 Agent 再补治理”，而是先把数据、规则、口径和责任链打稳。对本体和图数据库来说，关键价值在于把数据资产放入业务语义网络中：字段是什么意思，指标从哪里来，规则适用于谁，某个变更影响哪些报表、模型和业务流程。

### 2. 本体正在从知识描述变成运行时政策

传统本体用于统一概念、实体、关系和属性。Agent 时代，本体还应定义：

- 某类 Agent 可以执行哪些动作；
- 某类数据在哪些上下文下可读；
- 某个业务动作需要哪些前置条件；
- 哪些动作必须人工审批；
- 哪些结论必须保留证据；
- 出现规则冲突时如何决策。

结合图数据库，本体可以成为 Agent 的业务政策层。图数据库负责保存实例关系和执行链路，本体负责定义语义约束，Agent 负责基于上下文行动。

## 五、公司与竞品动态判断

### 1. OpenAI、ServiceNow、Oracle、EDB 都在争夺企业 Agent 控制权

OpenAI 通过 Presence 和 DeployCo 进入企业流程落地；Oracle 把 A2A 服务嵌入数据库；EDB 强调 Agentic Database；ServiceNow 通过 AI Control Tower 争夺企业 AI 治理入口。这些动作共同说明：Agent 时代的竞争不再是单点模型 API，而是“谁掌握企业上下文、工具、流程和治理控制面”。

对图数据库厂商而言，最危险的不是 Neo4j 或 TigerGraph，而是企业软件和云厂商把 Agent 控制平面做成平台入口。如果图数据库只是底层存储，很容易被平台吞掉；如果能成为关系治理和语义控制层，就能进入核心架构。

### 2. 图数据库厂商要同时讲 AI 能力和生产能力

Neo4j 的优势在 GraphRAG、知识图谱和开发者生态；TigerGraph 强调规模、企业安全、GraphRAG 和图分析。国产图数据库要进入金融、政务和大型企业，不能只讲性能，还要讲：

- 统一身份；
- 权限和审计；
- 备份恢复；
- 滚动升级；
- Kubernetes 和云原生；
- 本体融合；
- Agent 工具调用；
- GraphRAG 证据链；
- 决策与行动血缘。

## 六、对工作的可用要点

### 1. 产品定位建议

建议将图数据库定位从“复杂关系数据库”升级为：

> 企业 AI Agent 的可信关系与语义治理底座。

这句话能把图数据库、本体、AskGraph、MCP、GraphRAG、Dry Run、审计和金融行业方案统一起来。

### 2. 建议建设 Agent Governance Graph 标准模型

节点：User、Agent、Role、Permission、Tool、API、Dataset、Model、Prompt、Policy、BusinessObject、Workflow、Action、Evidence、RiskEvent、CostCenter。

关系：owns、delegates、can_access、calls、reads、writes、depends_on、governed_by、produces、affects、audited_by、violates、rollback_to。

关键查询：

- 某 Agent 为什么有权执行某动作？
- 某数据源变化影响哪些 Agent？
- 某工具被哪些业务流程调用？
- 某风险事件涉及哪些权限链？
- 某个决策依据哪些证据和规则？

### 3. GraphRAG 的销售话术应升级

不要只讲“召回更准”，而要讲：

> GraphRAG 让 AI 的上下文可解释、可裁剪、可验证、可审计。

尤其金融客户更关注证据链，而不是炫技式问答。

### 4. 面对 PostgreSQL SQL/PGQ 的话术

> SQL/PGQ 会让更多关系数据库用户理解图查询，这是图技术普及的积极信号；但企业级图数据库解决的是大规模关系网络上的高性能推理、图算法、路径分析、GraphRAG 和 Agent 运行时治理问题。

### 5. 金融行业切入点

优先选择监管报送、授信审查、反洗钱、模型治理和科技风险管理。这些场景都需要规则、证据、关系和审计，适合展示本体 + 图数据库 + Agent 的组合价值。
