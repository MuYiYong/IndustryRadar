# 每日行业简报｜2026-07-22

## 今日核心判断

**Agent 正在成为下一代企业软件入口，而图数据库、本体和数据治理正在从“辅助 AI”升级为“AI 时代的控制平面”。**

过去两年的企业 AI 竞争核心是“谁拥有更强的大模型”。进入 2026 年后，竞争焦点快速转向“谁能够让 Agent 在企业环境中安全理解业务、访问数据、调用工具、执行动作，并且全过程可追踪、可解释、可治理”。

未来企业 AI 架构很可能形成：模型负责推理，Agent 负责行动，本体负责理解业务，图数据库负责表达关系，数据治理负责约束可信边界。

---

## 一、Agent 基础设施：从工具调用走向运行时治理

### 1. Agent Runtime 进入可观测、可计量、可告警阶段

AWS Bedrock AgentCore 在 2026 年 7 月发布说明中新增 `ActiveSessionCount` 指标，让 Runtime、Code Interpreter 和 Browser 等服务可以在 CloudWatch 中发布活跃会话数。

参考：[Amazon Bedrock AgentCore Release Notes](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/release-notes.html)。

这个变化说明 Agent 已经成为一种新的企业运行时负载。企业未来会像管理微服务一样管理 Agent，但 Agent 比微服务更复杂，因为它具有动态规划、工具调用和不确定执行路径。

Agent 生产化至少需要：

1. 身份管理：Agent 是谁，代表谁。
2. 工具管理：Agent 可以调用什么。
3. 上下文管理：Agent 理解什么业务信息。
4. 运行观测：Agent 如何消耗资源。
5. 行动治理：Agent 是否在授权范围内执行。

图数据库最适合切入上下文和治理部分，因为这些问题本质上都是关系问题。

## 二、数据库进入 Agent 原生阶段

### 1. Oracle 将 A2A 服务嵌入 Autonomous AI Database

Oracle Autonomous AI Database 支持托管 Agent2Agent Server，可以通过开放 A2A 协议暴露 Select AI agent teams，使外部 Agent 能发现和调用数据库内 Agent，同时保持 Agent 逻辑和数据访问在数据库内部。

参考：[Expose Select AI Agent team as agent through A2A protocol](https://docs.oracle.com/en-us/iaas/releasenotes/autonomous-database-serverless/2026-07-expose-select-ai.htm)、[Oracle Autonomous AI Database A2A Server](https://blogs.oracle.com/developers/introducing-oracle-autonomous-ai-database-a2a-server-for-governed-multi-agent-systems)。

这个趋势说明数据库厂商正在争夺 Agent 数据入口。数据库不再只是 Agent 查询的数据源，而希望成为可信 Agent 节点。

但企业 Agent 不会只存在于一个数据库中，因此跨系统关系治理仍然需要图数据库和本体。

### 2. EDB 把 Agent 治理下沉到 Postgres 数据层

EDB Agentic Database 强调使用 Postgres 原生角色、行级安全和会话审计，把 Agent 身份、用途、权限和企业策略组合成受约束请求。

参考：[EDB Launches Agentic Database, Converged Analytics, and Governance](https://www.enterprisedb.com/press-releases/edb-launches-agentic-database-converged-analytics-and-governance-bringing-sovereign)。

数据库优势是可以在数据访问位置执行策略，但无法独立理解跨系统业务动作。

因此未来架构会形成：

- 数据库负责本地执行控制。
- 图数据库负责全局关系治理。
- 本体负责业务语义统一。

## 三、图数据库新定位：Enterprise Context Graph

### 1. 从 Knowledge Graph 到 Context Graph

论文《Context Graphs for Proactive Enterprise Agents》提出 Context Graph，用实时关系结构表达企业实体、关系和状态变化，使 Agent 可以主动发现信息，而不是等待用户提问。

参考：[Context Graphs for Proactive Enterprise Agents](https://arxiv.org/abs/2607.07721)。

传统知识图谱回答“世界是什么样”，Context Graph 进一步回答：

- 当前发生了什么变化？
- 变化影响谁？
- 下一步应该做什么？

Context Graph 包含：

- 实体关系；
- 当前状态；
- 状态变化；
- 行动上下文。

这使图数据库从知识管理平台升级为 Agent 上下文状态层。

### 2. 本体与 Context Graph 协同

本体负责定义概念、规则和业务口径；Context Graph 保存真实实例、状态变化和影响路径。

例如监管报送场景：

- 本体定义指标、字段、规则、机构和产品。
- 图数据库记录具体数据血缘、质量状态、规则触发和整改流程。
- Agent 完成规则发现和解释生成。

这比传统智能问答更接近企业真实需求。

## 四、GraphRAG 进入生产化阶段

### 1. GraphRAG 重点从召回转向上下文工程

《Is GraphRAG Needed?》研究比较普通 RAG、GraphRAG、Modular RAG 和 Agentic RAG，并指出高级检索不一定带来等比例生成质量提升，同时强调上下文工程的重要性。

参考：[Is GraphRAG Needed? From Basic RAG to Graph-/Agentic Solutions with Context Optimization](https://arxiv.org/abs/2606.25656)。

GraphRAG 未来价值不只是提升召回，而是：

- 让上下文结构化；
- 让答案可解释；
- 让证据可追踪；
- 让 Agent 可基于证据执行。

### 2. Neo4j 强化向量搜索工程能力

Neo4j 2026.06.0 引入高保真量化向量搜索预览能力，通过量化向量候选搜索和全精度重排降低内存消耗并提升效率。

参考：[Neo4j Release Notes](https://neo4j.com/release-notes/)、[Neo4j Vector Indexes](https://www.neo4j.com/docs/cypher-manual/25/indexes/semantic-indexes/vector-indexes/)。

图数据库向量竞争已经进入工程优化阶段：

- 向量成本；
- 混合查询；
- 图约束召回；
- 证据排序；
- 查询分析。

## 五、数据库与图数据库竞争判断

PostgreSQL 19 Beta 2 发布，SQL/PGQ 属性图能力继续推进。

参考：[PostgreSQL 19 Beta 2 Released](https://www.postgresql.org/about/news/postgresql-19-beta-2-released-3350/)。

这会推动更多用户理解图查询，但专用图数据库需要强调：

- 高性能多跳遍历；
- 路径分析；
- 图算法；
- 实时关系计算；
- Agent 治理图谱；
- GraphRAG 上下文工程。

SQL/PGQ 是图能力普及，而专用图数据库是关系智能生产基础设施。

## 六、金融科技、本体与数据治理

金融行业真正需要的是可信执行，而不是聊天机器人。

高价值场景包括：

### 风控

客户、账户、交易、设备、企业关系形成复杂网络，需要多跳推理。

### 监管报送

监管规则、指标、字段、系统、数据质量和报表之间需要统一语义和关系治理。

### 模型治理

模型、数据、特征、规则、审批和业务影响需要完整证据链。

建议金融 Agent 采用分级治理：

1. 只读型 Agent。
2. 建议型 Agent。
3. 受控执行型 Agent。
4. 高自治型 Agent。

不同等级对应不同权限、审计和人工确认要求。

### 1. 产品定位

从“图数据库”升级为：

> 企业智能关系基础设施。

核心价值是连接人员、数据、规则、流程、模型和 Agent。

### 2. Agent Governance Graph 标准模型

节点：

- User
- Agent
- Role
- Permission
- Tool
- API
- Dataset
- Model
- Prompt
- Policy
- BusinessObject
- Workflow
- Action
- Evidence
- RiskEvent

关系：

- owns
- delegates
- can_access
- calls
- reads
- writes
- depends_on
- governed_by
- affects

### 3. 金融方案表达

不要强调 AI 问答，应强调：

- 可信数据；
- 可信规则；
- 可信 Agent；
- 可信审计。

### 4. 对 PostgreSQL SQL/PGQ 的竞争话术

> SQL/PGQ 会推动图技术普及，但专用图数据库解决的是复杂关系智能生产问题，包括多跳遍历、图算法、实时关系计算和 Agent 治理。
