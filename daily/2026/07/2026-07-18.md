# 每日行业简报｜2026-07-18

## 今日核心判断

**企业 AI 正在从“模型驱动的功能增强”进入“Agent 驱动的运行时治理”阶段。**

今天最值得关注的信号不是某一个模型能力的提升，而是多个产业方向同时收敛到同一个问题：当 Agent 开始代表员工、系统或业务部门读取数据、调用工具、修改系统状态时，企业必须建立一个统一的 **Agent Control Plane（Agent 控制平面）**。这个控制平面需要管理 Agent 的身份、上下文、工具、权限、成本、动作、证据链和回滚机制。

ServiceNow 近期把 AI Control Tower 扩展为跨系统发现、观测、治理、保护和度量 AI 系统的产品，并与 Microsoft Agent 365、NVIDIA Enterprise AI Factory 等生态集成；这说明大型企业软件厂商正在把“治理”定位为 AI 时代的新入口，而不是附属能力。citeturn200322search3turn200322search4turn200322search0

从数据库和图技术角度看，这一变化非常关键：Agent 治理天然是一个关系问题。谁授权了哪个 Agent？Agent 能调用哪些工具？工具会影响哪些业务对象？某个动作依据了哪些数据、本体、规则和模型？一个外部供应商或支付服务商故障会影响哪些业务流程？这些问题很难只用表格、日志或向量检索解决，反而非常适合用图数据库、本体和数据治理能力共同建模。

## 一、AI 与 Agent：治理正在成为企业 AI 的主战场

### 1. 企业 AI 治理框架正在迅速过时

TechRadar 近期文章指出，许多企业 AI 治理框架仍停留在“防止员工把敏感数据输入公共模型”的阶段，但 Agent 已经把风险从“输出内容”扩展到“系统级访问与行动”。文章提出的关键问题包括：员工能否知道 AI 工具可以访问哪些系统、是否能集中撤销 Agent 访问权限、政策是否明确规定允许做什么，以及是否治理 Agent 的系统级行为。citeturn200322news49

这说明企业治理重心已经发生变化：

- 过去治理的是模型输出是否安全；
- 现在治理的是 Agent 是否有权执行某个动作；
- 过去关注数据是否泄露；
- 现在还要关注 Agent 是否通过一系列“单步合法”的调用完成了整体不被授权的业务流程；
- 过去审计的是用户操作；
- 现在要审计用户、Agent、模型、工具、数据和业务对象之间的完整行动链。

Agent 的风险不在于它“像人一样思考”，而在于它可以**以机器速度、跨系统、连续执行动作**。因此，治理不能依赖事后人工抽查，也不能依赖每个项目组自建审批流程，而必须变成平台级运行时能力。

### 2. “治理是新护城河”成为企业软件厂商共识

The Economic Times 今日报道 ServiceNow 高管观点：AI 时代企业竞争优势将从传统软件服务，转向 AI 系统的编排、治理和安全。该观点虽然带有厂商立场，但与 ServiceNow 今年多项产品动作一致：AI Control Tower、Context Engine、Workflow Data Fabric、MCP Registry、与 Microsoft Agent 365 和 NVIDIA 的治理集成都在强化同一叙事。citeturn200322news48turn200322search2turn200322search6

这对图数据库和本体厂商有直接启发：如果企业未来采购的是“Agent 治理与上下文平台”，那么图数据库不能只以 GraphRAG 组件身份出现，而应证明自己可以成为 Agent 控制平面中的关系引擎。

一个更完整的 Agent 控制平面至少包括五层：

| 层级 | 作用 | 图/本体机会 |
|---|---|---|
| 身份层 | 管理用户、Agent、工具和系统身份 | 建模委托关系、权限继承和责任主体 |
| 上下文层 | 管理数据来源、业务对象、本体版本和时间有效性 | 建立实体、关系、状态、证据链 |
| 策略层 | 判断动作是否允许、是否审批、是否拒绝 | 用本体表达业务规则，用图计算影响范围 |
| 执行层 | 控制工具调用、MCP Server、API 和外部系统 | 管理工具目录、依赖关系和风险等级 |
| 证据层 | 保存输入、计划、工具调用、规则命中和业务影响 | 形成 Decision & Action Lineage |

**工作可用观点：**

> Agent 时代，企业治理的对象不再只是数据和用户，而是“机器主体如何在业务语义和权限边界内行动”。

## 二、图数据库市场：GraphRAG 正在成为基础能力，控制价值才是下一阶段差异化

这些能力看起来没有 GraphRAG 热闹，却非常重要。它说明图数据库进入大型企业和金融核心环境时，客户真正关心的是：

- 是否能进入统一身份体系；
- 是否支持服务间双向认证；
- 是否支持云角色和短期凭证；

### 2. 图数据库竞争正在从“查询价值”走向“控制价值”

过去图数据库主要讲三类价值：多跳查询、图算法、关系洞察。GraphRAG 兴起后，增加了“让大模型回答更准、更可解释”的价值。但如果 GraphRAG、向量索引、自然语言查询都逐渐成为各厂商标配，下一阶段的差异化就会转向控制价值。

可以把图数据库价值分为三层：

| 价值类型 | 典型问题 | 代表能力 |
|---|---|---|
| 查询价值 | 谁和谁有关？路径是什么？ | 图查询、路径、模式匹配、GQL/Cypher |
| 分析价值 | 网络中谁最关键？风险如何传播？ | 社区发现、中心性、相似性、传播算法 |
| 控制价值 | 谁能对谁做什么？动作会影响谁？如何追责？ | 权限图、依赖图、策略图、证据链、行动模拟 |

Agent 时代真正新增的是第三层。一个企业可以用向量数据库和搜索引擎做 RAG，但它很难仅靠向量相似度判断“这个 Agent 是否有权冻结这个账户”“调用这个工具会影响哪些客户”“这次决策依据了哪个版本的本体和规则”。

**工作可用观点：**

> GraphRAG 解决“找得到”，图算法解决“看得深”，Agent Control Graph 解决“做得对、控得住、追得回”。

## 三、数据库与开源技术：Agent 工作负载让连接、状态和经验图成为基础设施问题

### 1. PostgreSQL 19 Beta 2 体现成熟开源生态优势

PostgreSQL 全球开发组于 2026 年 7 月 16 日发布 PostgreSQL 19 Beta 2。官方强调 Beta 版本用于功能预览和兼容性测试，不建议在生产环境运行，但鼓励开发者用典型应用负载进行测试，以帮助社区发现问题并提升正式版本稳定性。citeturn719709search1turn503778search2

这条消息对 AI 应用的意义不只是“数据库版本更新”。PostgreSQL 正在成为大量 Agent 应用默认的控制数据库，用于保存：

- 用户、任务、会话和状态；
- Agent 计划、中间结果和工具调用；
- JSON 结构化上下文；
- 向量及其元数据；
- 审批、补偿和审计记录；
- 业务系统中的确定性状态。

这对图数据库构成现实竞争。客户会自然地问：如果 PostgreSQL 加 JSONB、pgvector、全文检索、递归查询和连接池已经能解决 80% 的问题，为什么还要引入专业图数据库？

图数据库必须用普通组合方案难以低成本解决的问题证明必要性，例如：

- 多级供应链依赖和替代路径；
- 跨账户、设备、IP、商户、客户的欺诈网络；
- 用户、Agent、工具、数据和策略之间的动态授权；
- 外部服务故障对业务流程的级联影响；
- 决策依据、执行动作和业务影响之间的多跳证据链。

### 2. Odyssey 1.5.1 提醒我们：Agent 首先会打爆连接与并发治理

Odyssey 1.5.1 于 2026 年 7 月 14 日发布。它是面向 PostgreSQL 和 Apache Cloudberry 的多线程连接池，新版本增加共享池、事务池中的 LISTEN 固定连接、负载均衡、CPU 亲和性、AWS-LC 构建、SSLKEYLOGFILE、队列等待时间指标、Prometheus 指标改进和扩展协议修复。citeturn719709search2

这不是一条典型 AI 新闻，但对 Agent 工程非常重要。Agent 会产生大量短生命周期、并发波动明显、工具链复杂的数据库请求。一个用户问题可能被拆成多个计划步骤，每个步骤又可能调用不同数据库和外部工具。如果连接、认证、队列和取消机制不稳定，系统会先在基础工程层面失败，而不是在模型能力上失败。

对图数据库也一样。自然语言查询和 Agent 工具调用通常比手写查询更不可预测，需要重点设计：

- Agent / 用户 / 租户级并发配额；
- 长路径查询和图算法的资源保护；
- 查询队列、等待时间和取消机制；
- 会话状态清理和连接复用安全；
- 工具调用链路的可观测性。

### 3. Experience Graphs：Agent 经验正在成为可查询的数据库资产

近期论文《Experience Graphs: The Data Foundation for Self-Improving Agents》提出，长周期 Agent 在代码生成、科学发现、硬件设计等任务中会不断生成工具输出、失败记录、分支、修复、奖励和因果链路，这些不应只是一次性日志，而应成为可治理、可查询、可恢复、可复用的数据库状态。论文把这种结构称为 Experience Graph，并提出 Trellis 作为数据基础。citeturn719709academia51

这对图数据库非常有启发。传统日志回答“发生了什么”，Experience Graph 更强调：

- Agent 当时知道什么；
- 走过哪些分支；
- 哪些尝试失败了；
- 哪个工具输出导致了下一步选择；
- 哪些经验可以跨任务复用；
- 哪些经验可以转化为训练数据。

**工作可用观点：**

> Agent 的长期价值不只在于完成一次任务，而在于把每次探索、失败、修复和成功沉淀为可查询、可复用、可审计的经验图。

## 四、本体与数据治理：从“语义解释”升级为“运行时约束”

### 1. Agent 需要确定性治理层，而不是只依赖模型对齐

AgentBound 论文提出，自治 Agent 正在执行金融交易、外部沟通和企业工作流等重要动作，现有身份联邦和授权机制只能证明 Agent 是否有访问权限，却无法判断某个动作在当前行为和业务上下文下是否应该执行。AgentBound 使用三类独立权威：委托授权、所有者签名的行为宪章、站点动作合同，对每个动作做 permit/review/deny 判断，并生成可验证的治理收据。citeturn200322academia54

这与本体和图数据库的结合点非常明确：

- 委托授权可以用图表达用户、角色、Agent 和工具之间的关系；
- 行为宪章可以由本体和策略定义；
- 站点动作合同可以描述工具参数、前置条件、风险等级和审批要求；
- 治理收据可以成为决策与行动血缘的一部分。

另一篇关于生产 Agent 运行时治理的五平面参考架构提出，Agent 风险不再只是跨越数据边界，而是进入工作流内部：Agent 读取上下文、调用工具、使用连接器并修改系统记录。它强调组合主体、能力衰减、stop-anywhere mediation 和结构化证据审计。citeturn200322academia53

### 2. 本体的角色必须从“业务词典”升级为“机器可执行政策”

过去本体常被理解为统一客户、产品、合同、账户、指标等业务术语。Agent 时代，本体还要表达：

- 某类主体可以执行哪些动作；
- 某类数据受到哪些法规或内部策略约束；
- 某个业务动作需要满足哪些前置条件；
- 哪些动作必须走人工审批；
- 哪些结果必须保留证据；
- 出现冲突时规则优先级如何判断。

因此，本体应该从“描述世界是什么”，升级为“规定机器在业务世界中可以怎样行动”。

**工作可用观点：**

> 企业本体的下一阶段价值，不是让大模型多知道一些术语，而是把业务对象、规则、权限和动作边界变成 Agent 运行时必须遵守的机器可执行政策。

## 五、金融科技与数字欧元：可编程支付推动关系治理进入工程阶段

ECB 同时在 2026 年 7 月发布数字欧元 scheme rulebook v0.91 及大量附件，内容覆盖测试认证、用户旅程、端到端流程、前端实现、后端异常管理、访问管理、风险与欺诈、争议管理、数据交换、结算和数据字典等。citeturn719709search0turn719709search3

### 2. 数字货币场景的图价值不是“保存交易”，而是“解释关系和责任”

数字欧元、稳定币和可编程支付都会带来复杂关系问题：

- 用户、钱包、账户、设备和商户之间的身份映射；
- PSP、收单机构、银行、央行平台之间的依赖关系；
- 线上与线下支付状态同步后的冲突处理；
- 风险地址、设备、商户和最终受益人之间的多跳关联；
- 争议、退款、异常恢复和证据之间的责任链；
- 某 PSP、云服务或支付通道故障时的业务影响范围。

这些都是图数据库和本体可以发挥作用的地方。特别是在金融行业，客户最终要回答的不只是“这笔交易是否可疑”，还包括“为什么可疑、关联到谁、违反了哪条规则、谁负责处理、处理后影响什么”。

**工作可用观点：**

> 可编程支付越成熟，金融基础设施越需要关系穿透、规则验证、风险传播分析和端到端证据链。

## 六、公司与竞品动态判断

### 判断一：ServiceNow 正在把 AI 治理做成企业入口

ServiceNow 的 Context Engine、Workflow Data Fabric、AI Control Tower、MCP Registry 和与 Microsoft、NVIDIA、Google Cloud 的合作，说明它并不满足于做某个 Agent，而是在争夺“所有 Agent 的控制平面”。citeturn200322search2turn200322search3turn200322search6turn200322search7

这对图数据库厂商是一个提醒：未来企业 AI 预算不一定流向“最会做 RAG 的数据库”，而可能流向“最能治理 Agent 行动的平台”。图数据库要进入核心预算，需要把自己定位为控制平面中的关系与语义基础设施。

### 判断二：PostgreSQL 生态继续形成“足够好且低复杂度”的竞争压力

PostgreSQL 的优势不是每项能力都最强，而是事务、JSON、扩展、工具、连接池、监控和人才生态形成了低组织成本。对于很多 AI 应用，PostgreSQL 组合方案会成为默认起点。专业图数据库必须说明自己不是替代所有业务数据库，而是解决普通组合方案难以低成本处理的复杂关系推理、影响分析和证据链问题。

### 判断三：图数据库厂商的短板正在从查询语言转向企业完整性

Neo4j、TigerGraph 等厂商都已进入“图 + 向量 + AI”阶段。接下来客户会比较：统一身份、mTLS、SSO、审计、备份恢复、Kubernetes、资源隔离、权限传播、决策血缘、Agent 工具治理等能力。谁能把图技术和企业级治理结合得更完整，谁就更容易进入金融、政务、运营商和大型制造的关键系统。

## 七、对悦数图的具体建议

### 建议一：打造 Agent Control Graph 产品原型

建议建立一个最小可用模型，至少包括：

- User；
- Agent；
- Model；
- Tool；
- MCP Server；
- Data Asset；
- Business Object；
- Policy；
- Action；
- Evidence；
- Impact。

最小演示可以回答三个问题：

1. 某个 Agent 为什么有权执行这个动作？
2. 某次动作使用了哪些数据、本体、规则、模型和工具？
3. 如果某个工具、数据源或外部服务不可用，会影响哪些 Agent 和业务流程？

这可以把图数据库从“查询数据关系”提升到“治理机器行动关系”。

### 建议二：将 OntoBuild 扩展为 Ontology-to-MCP / Ontology-to-Tools

OntoBuild 不应只输出本体类、属性和关系，还可以进一步生成：

- 查询工具；
- 创建和修改工具；
- Dry Run 工具；
- 参数类型和必填字段；
- 允许关系和禁止关系；
- 前置条件和审批规则；
- 风险等级和审计要求。

这样可以形成闭环：

> 本体定义 → 工具生成 → Agent 调用 → 图上验证 → Dry Run 模拟 → 正式执行 → 证据反馈 → 本体演化

### 建议三：把 Dry Run 从查询验证升级为行动模拟

Dry Run 可以分三层：

| 层级 | 说明 |
|---|---|
| Query Dry Run | 验证语法、执行计划、资源消耗、结果规模 |
| Data Dry Run | 预测将读取、修改或影响哪些数据对象 |
| Business Action Simulation | 在隔离图中模拟动作后的业务状态、规则命中、风险传播和回滚路径 |

这对金融行业尤其有价值。例如账户冻结、授信调整、交易拦截、供应商替换、客户风险评级变更等，都可以在真正执行前先模拟影响范围。

### 建议四：加强企业级安全能力的市场表达

TigerGraph 4.3 把 mTLS、Kerberos、SAML/OIDC、JWT Claim、AWS Role ARN 等能力放到版本重点中，说明安全能力正在成为图数据库进入关键系统的门槛。悦数图面向金融客户时，建议把以下能力与 Agent 场景绑定表达：

- Agent 独立身份；
- 用户权限向 Agent 的受控委托；
- 图查询、图算法和数据修改的分级授权；
- 短期凭证和证书认证；
- 全链路审计；
- Kill Switch 与紧急吊销；
- 敏感动作 Dry Run 与人工审批。

## 八、今日可直接用于工作的关键观点

### 观点 1：Agent 治理不是应用功能，而是基础设施

> 当 Agent 能够跨系统读取数据、调用工具并修改业务状态时，治理必须内置在身份、上下文、工具和执行链路中，而不是靠项目组事后补日志。

这句话适合用于说明为什么企业 AI 需要平台化治理，而不是每个应用自己做一个审批按钮。

### 观点 2：图数据库的 AI 价值不只是 GraphRAG

> GraphRAG 让模型找到更相关的信息；Agent Control Graph 让企业知道机器为什么能做、能做什么、做了什么、影响了谁。

这句话可以直接用于区分“图增强检索”和“图驱动治理”。

### 观点 3：本体是 Agent 的业务政策层

> 本体不仅定义客户、账户、合同和产品是什么，还应定义 Agent 在什么条件下可以查询、修改、提交、审批或拒绝某个业务动作。

这能把本体从知识管理项目升级为运行时治理能力。

### 观点 4：PostgreSQL 是默认起点，图数据库必须证明复杂关系价值

> PostgreSQL 生态的优势是低复杂度和成熟工具链；专业图数据库的机会，是解决普通组合方案难以低成本处理的多跳依赖、权限传播、风险扩散和行动证据链。

这能帮助销售和产品避免陷入“替代关系数据库”的错误叙事。

### 观点 5：数字货币首先是跨机构关系工程

> 数字欧元和稳定币的难点不只是账本，而是身份、钱包、PSP、商户、设备、规则、争议和结算之间的关系协同。

这为金融图数据库方案提供了新的切入点。

## 结论

今天的产业主线可以概括为：**Agent 正在成为企业中的新型机器主体，而治理正在成为企业 AI 的核心控制面。** ServiceNow 等企业软件厂商正在争夺 AI Control Tower 入口；PostgreSQL 生态继续以成熟、低复杂度和工程工具链承接 AI 工作负载；Neo4j、TigerGraph 等图数据库厂商则在 GraphRAG、向量搜索、企业安全和场景化选型中持续推进；数字欧元进入试点工程准备阶段，进一步说明金融基础设施将越来越依赖跨机构关系建模和证据链。citeturn200322search3turn719709search1turn719709search6turn503778search1

对悦数图而言，最值得强化的方向不是简单说“图数据库 + 大模型”，而是形成更强的产品主线：

> **以图数据库计算身份、数据、工具和业务对象之间的关系，以本体定义机器可执行规则，以 MCP 连接 Agent 与企业工具，以 Dry Run 和证据链保证每次行动可验证、可控制、可追责。**

这条主线能把 OntoBuild、AskGraph、图算法、MCP、Dry Run、数据治理和企业安全统一起来，也更符合企业 AI 从 Demo 走向生产后的真实采购逻辑。
