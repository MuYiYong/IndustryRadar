## 一、AI 与 Agent：治理需要从“人工兜底”升级为“架构内生”

近期关于大规模 Agent 治理的分析提出，到 2028 年，部分大型企业可能运行十万量级的 Agent，而目前只有约 12% 的企业拥有集中式治理方式。即便具体预测需要谨慎看待，其揭示的结构性矛盾是成立的：企业正在以项目为单位快速创建 Agent，但身份、工具、知识、日志和策略仍分散在不同平台中。citeturn140341news49

### 1. Human-in-the-loop 不能替代统一治理

人工审批适合高风险、低频操作，但无法覆盖数千个 Agent 的日常调用。若每个团队自行设计审批流程，会出现：

- 同一数据在不同 Agent 中采用不同访问标准；
- 相似动作被划分为不同风险等级；
- Agent 使用共享账号，责任主体无法区分；
- 审批只看到最终动作，看不到上下文和前置调用；
- 大量低风险动作排队等待人工确认，系统无法规模化。

因此，Human-in-the-loop 应是治理体系的一部分，而不是治理体系本身。生产级 Agent 更需要“policy-in-the-loop”：在每次调用和执行时，由统一策略层自动判断主体、对象、环境、风险和动作是否匹配。

### 2. 企业需要 Agent Control Plane

比较完整的控制平面应至少覆盖五类对象：

| 对象 | 需要治理的内容 |
|---|---|
| Agent | 所有者、用途、模型、版本、风险等级、生命周期 |
| Context | 数据来源、本体版本、时间有效性、敏感等级 |
| Tool | 可执行动作、参数、外部依赖、调用成本、失败影响 |
| Policy | 权限、审批、地域、时间、客户类型和监管限制 |
| Evidence | 输入、计划、工具调用、规则命中、输出和业务影响 |

**工作可用观点：**

> Agent 数量增加后，治理重点不再是审核每一次回答，而是把身份、权限、上下文和动作规则做成默认执行的系统能力。

## 二、图数据库市场：竞争焦点转向“场景适配与企业完整性”

TigerGraph 7 月 17 日发布的 2026 图数据库选型指南，把 AI Agent、企业级应用、反欺诈、定价和架构权衡列为主要比较维度。这类厂商内容带有明显营销立场，不能视为中立评测，但它反映出市场销售话术已经变化：客户不再只问“哪个图数据库更快”，而是问“哪个产品适合我的 AI、风控、云架构和组织约束”。citeturn140341search5

### 1. GraphRAG 正在成为入场券，而不是长期壁垒

Neo4j 当前最新数据库版本仍是 2026.06.0，其重点之一是 Hi-Fidelity Quantized Vector Search：先在量化向量上扩大候选召回，再用全精度向量重新排序，以降低内存和查询成本并改善召回。TigerGraph 4.2 则已把向量属性、ANN 索引和图向量混合查询放进 GSQL。citeturn140341search1turn140341search2

这说明图数据库主要厂商已经形成共同技术基线：

- 支持向量属性和索引；
- 支持向量候选与图模式组合；
- 支持 GraphRAG 或知识图谱增强检索；
- 支持自然语言或 Agent 框架集成。

因此，单纯宣传“图 + 向量”很难持续差异化。下一阶段需要回答的是：

1. 图和向量是否在同一查询计划中协同，而非应用层拼接；
2. 检索结果是否携带来源、时间、版本和权限；
3. Agent 是否能够安全地从查询进入业务动作；
4. 大规模图计算是否可以与在线服务协同；
5. 产品是否具备企业身份、安全、备份、升级和审计能力。

### 2. 图数据库选型将分成三类价值

| 价值类型 | 客户核心问题 | 典型能力 |
|---|---|---|
| 查询价值 | 能否高效回答复杂关系问题 | 多跳、路径、模式匹配、GQL/Cypher |
| 分析价值 | 能否发现网络中的结构和风险 | 社区、中心性、传播、相似性、连通性 |
| 控制价值 | 能否约束并解释机器行动 | 权限图、依赖图、策略判断、影响模拟、证据链 |

传统图数据库营销主要覆盖前两类，而 Agent 时代正在创造第三类预算。对悦数图而言，最值得强化的不是泛化为另一个 Agent 平台，而是把图查询和图算法升级为 **Agent 权限、依赖和行动影响的关系引擎**。

## 三、数据库与开源生态：连接管理重新成为 AI 应用的基础问题

Odyssey 1.5.1 于 7 月 14 日发布。它是面向 PostgreSQL 和 Apache Cloudberry 的多线程连接池，新版本增强了共享池、事务池中的 LISTEN 固定连接、负载均衡、CPU 亲和性、AWS-LC 构建、排队时延指标、Prometheus 指标和扩展协议兼容性。citeturn186522search0

这不是一条“耀眼”的 AI 新闻，却具有很强的工程意义。Agent 会产生大量短生命周期、并发度波动明显的数据库请求：一个用户任务可能被拆分为多个计划步骤，并由多个工具并行访问数据库。若每个步骤都直接建立连接，数据库很容易先在连接数、认证和调度上遇到瓶颈，而不是在 SQL 执行本身遇到瓶颈。

### 1. Agent 工作负载会放大连接池的四个问题

- **突发性：** Agent 规划后可能同时触发多条查询；
- **长短事务混合：** 普通读取、流式结果和工具事务并存；
- **租户隔离：** 不同 Agent、用户和项目需要独立配额；
- **可观测性：** 需要知道等待时间、池占用和后端失败，而不只是查询耗时。

Odyssey 1.5.1 增加平均等待时间、总等待时间和等待提示，说明连接层可观测性正在变得更重要。其 `pool_pin_on_listen`、失败后端透明关闭和扩展协议修复，也反映出事务池不能只追求复用率，还必须正确处理状态型协议行为。citeturn186522search0

### 2. 对图数据库的启发

图数据库面向 Agent 和自然语言查询时，也需要把连接与会话治理纳入产品设计：

- 每个 Agent 是否拥有独立身份和连接额度；
- 长路径查询是否占满执行资源；
- 连接池是否保持错误的 session 状态；
- 流式查询中断后是否及时释放服务端资源；
- 查询队列、等待时间和取消操作是否可观测。

**工作可用观点：**

> Agent 应用的数据库瓶颈，往往先出现在连接、并发、身份和资源隔离，而不是模型或查询语言本身。

- 离线交易恢复后的冲突和风险传播；

- 规则、身份、交易和审计证据之间的关联追溯。

## 五、本体与数据治理：从统一词义走向统一控制语义

大规模 Agent 治理的核心难题，是不同团队对“客户”“敏感数据”“高风险动作”“人工审批”等概念理解不一致。仅靠权限表无法解决，因为权限判断往往依赖业务语义：同一字段在不同业务场景中的敏感等级可能不同，同一动作对普通客户和高风险客户的审批要求也不同。

### 1. 本体需要覆盖四类语义

1. **对象语义：** 客户、账户、合同、Agent、工具分别是什么；
2. **关系语义：** 谁拥有谁、谁委托谁、谁依赖谁、谁影响谁；
3. **规则语义：** 什么条件下允许、拒绝或要求审批；
4. **行动语义：** 查询、建议、提交、修改、支付等动作造成什么业务后果。

本体只有进入策略判断和工具调用环节，才会从“高质量词汇表”变成 Agent 的运行时基础设施。

### 2. 数据治理要增加 Agent 资产目录

企业现有数据目录通常管理表、字段、报表、指标和负责人。未来还应登记：

- Agent 的业务目标和责任人；
- 使用的模型、数据、本体和工具；
- 可执行动作及风险等级；
- 服务级别、成本和调用配额；
- 审批人、Kill Switch 和回滚方式；
- 历史事故和策略变更记录。

这使数据治理从管理“可被使用的数据”，扩展到管理“使用数据并改变业务状态的机器主体”。

## 六、公司与竞品判断

### 判断一：图数据库厂商正在同时争夺 AI 预算和关键系统预算

Neo4j 强化向量搜索、GraphRAG 和 Connected Intelligence；TigerGraph 强调规模化图分析、混合向量检索和企业场景；PostgreSQL 生态则通过成熟事务、JSON、扩展、连接池和运维工具构成“足够好且低复杂度”的组合。citeturn140341search1turn140341search2turn186522search0

图数据库不能只和图数据库比较。真正的竞争通常是：

> 专业图数据库方案 vs PostgreSQL + 向量扩展 + 搜索 + Agent 框架的组合方案。

### 判断二：市场将更重视总拥有成本，而不是单项性能

客户会综合计算：

- 新增数据库带来的运维和人才成本；
- 数据同步及一致性成本；
- 身份、安全和审计集成成本；
- 查询开发和业务建模成本；
- 故障恢复、升级和扩容成本；
- 复杂关系问题未被解决带来的风险成本。

因此，专业图数据库必须证明其不仅查询更自然，还能减少应用层拼装、规则复制和影响分析的复杂度。

### 判断三：Agent 治理可能成为本体和图数据库的新共同市场

本体适合定义业务对象和规则，图数据库适合保存实例关系、计算传播路径和解释权限来源。二者结合，可以形成普通 IAM、向量库和关系库难以独立提供的能力：

- 动态授权；
- 多跳依赖；
- 上下文策略；
- 行动影响模拟；
- 决策与行动血缘。

## 七、对悦数图的具体建议

### 建议一：设计“Agent Control Graph”最小可用模型

建议至少包含：User、Agent、Model、Tool、DataAsset、BusinessObject、Policy、Action、Evidence、Impact 十类对象，并展示三个查询：

1. 某 Agent 为什么有权执行这个动作；
2. 某数据或工具故障会影响哪些 Agent 和业务；
3. 某次动作使用了哪些上下文、规则和证据。

### 建议二：把 AskGraph 的并发和连接治理产品化

自然语言查询通常比手写查询更不可预测，应增加：

- Agent / 用户级并发和资源配额；
- 查询等待时间与取消机制；
- 高风险或高成本查询的 Dry Run；
- 长路径、全图扫描和图算法的资源保护；
- 连接、会话和执行链路的统一可观测性。

### 建议三：把数字货币方案从“交易追踪”扩展到“支付生态关系图”

除钱包和交易关系外，加入 PSP、银行、商户、设备、认证、规则、基础设施和服务依赖，形成身份风险、支付风险和运营韧性三类能力。

### 建议四：市场材料同时回答技术和采购问题

对外比较材料不应只列性能、语法和算法数量，还应回答：

- 为什么不能只用 PostgreSQL；
- 图数据库会增加多少运维复杂度；
- 如何和现有关系库、数据湖协同；
- 如何进入企业 SSO、审计和备份体系；
- 哪些风险或决策问题只有图能低成本解决。

## 八、今日可直接用于工作的关键观点

### 观点 1：Agent 治理必须集中，执行可以分布

> 企业可以允许不同团队创建和运行 Agent，但身份、策略、工具目录、审计和紧急停止能力必须统一。

分布式创新能够提高速度，集中式控制则避免同一企业内部形成互不兼容的安全和业务规则。

### 观点 2：图数据库应从“关系查询引擎”升级为“关系控制引擎”

> 图不仅用来发现谁与谁有关，还应判断谁能对谁做什么、一个动作会影响谁，以及责任如何沿关系链追溯。

这比单纯 GraphRAG 更接近企业 Agent 的核心生产需求。

### 观点 3：数据库生态竞争本质上是复杂度竞争

> PostgreSQL 生态的优势并非每项能力都最强，而是能够用成熟工具和较低组织成本覆盖大量需求。

专业图数据库必须用复杂关系推理、传播分析和可解释控制，证明新增组件的收益大于系统复杂度。

### 观点 4：数字货币首先是一项跨机构系统工程

> 真正困难的不是创建一个数字余额，而是让央行、银行、PSP、商户、用户、设备和监管规则在同一支付流程中可靠协同。

这正是关系建模、身份解析和依赖分析能够发挥价值的地方。

## 结论

今天的产业信号可以归纳为一句话：**Agent 正在从应用功能变成企业中的新型机器主体。** 当 Agent 数量增长、调用链拉长、工具权限扩大后，依靠单项目审批和人工兜底无法形成规模化生产能力。企业需要统一的 Agent 控制平面，而图和本体可以分别承担关系计算与语义约束。

图数据库市场也因此进入新的竞争阶段。向量搜索和 GraphRAG 已逐渐成为基础能力，真正有价值的差异将来自企业安全、运行治理、依赖分析、动态授权和行动影响模拟。PostgreSQL 生态继续以低复杂度和完整工具链形成强大竞争压力，专业图数据库必须明确自己解决的是哪些普通组合方案难以低成本解决的关键关系问题。

对悦数图而言，值得持续强化的产品主线是：

> **以图数据库计算身份、数据、工具和业务对象之间的关系，以本体定义机器可执行规则，以 AskGraph 和 MCP 连接 Agent，以 Dry Run 和证据链保证每次行动可验证、可控制、可追责。**
