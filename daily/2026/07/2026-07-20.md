# 每日行业简报｜2026-07-20

## 今日核心判断

**企业 AI 正在从“在数据库旁边外挂智能”转向“把 Agent 身份、权限、策略和审计直接下沉到数据层”。**

过去两年，企业 AI 的主流架构通常是：数据存放在数据库或湖仓中，外部应用通过 RAG、向量检索或工具调用访问数据，再由大模型生成结果。最新一轮数据库厂商的动作表明，这种“数据在内、智能在外”的架构正在发生变化：数据库开始直接提供 Agent 接入、身份映射、细粒度授权、协议服务、向量检索和审计能力。

这并不意味着所有 Agent 都会运行在数据库内部，而是意味着数据库正在争夺 Agent 时代最关键的控制点：**谁能够在数据被读取、组合和修改的瞬间执行策略，谁就更接近企业 AI 的治理入口。**

对图数据库、本体和数据治理厂商而言，这既是竞争压力，也是机会。关系数据库会强化数据层治理，但跨系统的业务语义、复杂依赖、多跳影响、权限传递和行动证据链仍然需要图与本体来表达。未来更合理的企业架构，很可能是“数据库负责本地强制执行，图与本体负责跨系统语义和全局治理”。

## 一、数据库进入 Agent 原生阶段：治理开始下沉到数据访问的最后一公里

### 1. EDB 把 Agent 治理直接嵌入 PostgreSQL 权限体系

EnterpriseDB 在 EDB Postgres AI 中推出 Agentic Database、融合分析和治理能力，明确提出：Agent 的访问控制不应只依靠数据库之外的控制平面，而应在数据所在位置执行。其预览方案把 Agent 身份、声明用途、权限和企业策略组合为受约束的数据库请求，并利用 PostgreSQL 角色、行级安全和会话审计进行强制执行。

参考：[EDB Launches Agentic Database, Converged Analytics, and Governance](https://www.enterprisedb.com/press-releases/edb-launches-agentic-database-converged-analytics-and-governance-bringing-sovereign)。

这个方向的重要性不在于“数据库也能调用大模型”，而在于数据库厂商开始重新定义 Agent 的安全边界。传统中间层可以决定某个 Agent 是否允许访问某个 API，但只有数据库最清楚最终访问了哪些表、哪些行、执行了什么操作。因此，数据层治理具有三个天然优势：

- 策略执行位置离数据最近，不容易被旁路；
- 可以复用成熟的角色、行级权限、事务和审计机制；
- 能够把 Agent 与人类用户放在同一套数据安全体系中管理。

但这套模式也有明显边界：单个数据库只能看到本地数据访问，难以理解一个 Agent 跨多个系统完成的整体业务动作。例如，某个 Agent 先查询客户信息，再调用外部评分服务，随后修改授信额度；每一步单独看都可能合法，但组合起来是否符合业务授权，需要跨系统关系和流程语义才能判断。

因此，一个更完整的架构应当分为两层：

1. **本地执行层**：数据库基于角色、行级权限、事务和策略阻止越权操作；
2. **全局治理层**：图数据库与本体描述 Agent、用户、工具、数据、业务对象、规则和流程之间的关系，判断多步行动是否合法。

### 2. Oracle 将 A2A 服务嵌入 Autonomous AI Database

Oracle 发布 Autonomous AI Database Agent-to-Agent Server，使兼容 A2A 协议的客户端能够发现和调用基于 Select AI Agent 构建的企业 Agent，并把服务直接嵌入 Autonomous AI Database Serverless 和 Dedicated Region 环境。用户无需单独部署 A2A 服务基础设施。

参考：[Introducing Oracle Autonomous AI Database A2A Server for Governed Multi-Agent Systems](https://blogs.oracle.com/machinelearning/introducing-oracle-autonomous-ai-database-a2a-server-for-governed-multi-agent-systems)。

这个动作说明数据库的产品边界正从“存储与查询”扩展到“Agent 注册、发现、调用和协作”。当 A2A、MCP 等协议逐渐普及，数据库厂商希望把自己变成企业 Agent 网络的可信入口，而不仅仅是被 Agent 调用的数据源。

Oracle 的优势是能够把 Agent 服务与数据库身份、数据权限和企业部署环境紧密结合；风险则是企业可能形成新的平台绑定。对独立数据平台和图数据库厂商来说，产品设计应尽量兼容开放协议，同时避免把业务语义和 Agent 目录锁定在某个模型或云平台中。

一个值得借鉴的产品方向是建设 **Agent Registry Graph**：

- 记录每个 Agent 的所有者、用途、版本、风险等级和服务地址；
- 记录 Agent 可调用的工具、可访问的数据和可影响的业务对象；
- 记录 Agent 之间的委托、协作、依赖和调用链；
- 支持从业务对象反查相关 Agent，从异常行动追溯完整责任链。

这比简单的 Agent 列表更有价值，因为企业真正要管理的是关系和影响，而不是数量。

## 二、图数据库的新机会：从知识图谱升级为 Context Graph

### 1. Context Graph 正在成为主动式企业 Agent 的基础结构

近期论文《Context Graphs for Proactive Enterprise Agents》提出一种实时关系结构，用于描述企业实体、关系和状态变化，并通过变化检测、主动性评分和内容生成机制，在用户提出问题之前主动推送相关信息。论文在合同管理、工程事件响应和销售管道等案例中展示了基于上下文图的主动发现能力。

参考：[Context Graphs for Proactive Enterprise Agents](https://arxiv.org/abs/2607.07721)。

这个概念值得图数据库行业重点关注。传统知识图谱通常回答“世界是什么样”，而 Context Graph 还要回答“现在发生了什么变化”“这个变化对谁重要”“下一步应该做什么”。它将图从静态知识容器变成持续更新的业务状态网络。

Context Graph 至少包含四类信息：

- **实体与关系**：客户、合同、订单、设备、项目、人员及其业务关联；
- **当前状态**：风险等级、审批阶段、库存水平、服务状态等；
- **状态变化**：新增、删除、数值变化、关系变化和时间事件；
- **行动上下文**：哪些变化需要通知谁，可以触发什么规则或 Agent。

这与本体的结合非常自然。本体负责定义业务对象、状态、规则和动作语义；图数据库负责保存动态关系和变化链；Agent 负责分析、解释和执行。三者组合后，企业 AI 才能从“被动问答”走向“主动发现与行动”。

### 2. 对图数据库产品定位的启示

图数据库厂商不应只把 GraphRAG 描述为“比向量检索多一层关系”。更有竞争力的定位是：

> 图数据库是企业 Agent 的上下文状态层，用于维护业务对象之间的关系、状态变化、行动依赖和证据链。

这个定位能够同时连接多个市场需求：

- GraphRAG：为回答提供结构化上下文和多跳证据；
- Agent Memory：保存长期任务状态、交互历史和业务关系；
- Agent Governance：描述权限、工具、策略和影响范围；
- Proactive Agent：基于关系和状态变化主动发现机会与风险；
- Digital Twin：以图方式表达业务对象及其实时状态。

对悦数这类图数据库产品而言，可以考虑把“Context Graph”沉淀为一套参考架构和演示模型，而不是仅作为概念宣传。可以选择金融监管报送、供应链风险或系统运维作为样板场景，展示从数据变化到影响分析、Agent 判断和行动审计的完整链路。

## 三、图数据库竞品：Neo4j 强化向量检索效率，图与向量融合进入工程优化阶段

Neo4j 2026.06.0 引入高保真量化向量搜索预览能力。其思路是先在量化向量上扩大候选搜索范围，再使用全精度向量重新排序，以降低内存需求、提升查询速度并维持较高召回质量。

参考：[Neo4j Release Notes](https://neo4j.com/release-notes/)。

这个变化说明图数据库中的向量能力已经从“是否支持”进入“如何降低成本并稳定生产效果”的阶段。早期图向量融合主要解决功能完整性：图数据库需要支持 embedding 存储、向量索引和混合查询。现在竞争重点逐渐转向：

- 向量索引需要占用多少内存；
- 大规模数据下召回率与性能如何平衡；
- 图过滤和向量搜索的执行顺序如何优化；
- 是否支持二阶段召回与重排；
- 是否能够统一观察图查询和向量查询的成本。

对产品规划而言，仅提供 HNSW 或相似度函数已经不足。下一阶段需要重点考虑：量化、过滤下推、批量向量查询、图约束下的候选裁剪、混合代价估算以及冷热分层。

一个更有差异化的方向是 **Graph-guided Vector Search**：先利用业务关系、权限、本体类别或时间范围缩小候选图，再进行向量检索；或者先用向量发现候选实体，再通过路径、规则和关系强度重排。这样才能真正体现图数据库与独立向量库的差异。

## 四、开源与数据库趋势：PostgreSQL 正在把图、向量和 Agent 能力吸收到统一平台

PostgreSQL 生态正在同时吸收 SQL/PGQ、向量检索、AI 工具和 Agent 数据治理能力。PostgreSQL 19 Beta 2 已于 7 月 16 日发布，虽然测试版本不适合直接用于生产，但它进一步强化了市场对 PostgreSQL 作为统一数据与 AI 平台的预期。

参考：[PostgreSQL Release Notes](https://www.postgresql.org/docs/release/)、[PostgreSQL 19 Beta 1 Released](https://www.postgresql.org/about/news/postgresql-19-beta-1-released-3313/)。

这一趋势对专用数据库厂商意味着两件事。

第一，通用数据库会不断扩大能力边界。图查询、向量搜索、全文检索和 AI 接口会逐渐成为通用数据库的标准能力，客户会更频繁地问“为什么不能直接使用 PostgreSQL”。

第二，专用数据库必须证明自己不仅提供一种语法或索引，而是能在关键工作负载上提供数量级差异。对于图数据库，这些工作负载包括：

- 高并发、多跳、可变长度路径查询；
- 大规模图算法和增量图计算；
- 复杂关系网络中的模式匹配和影响分析；
- 分布式图存储、跨分区遍历和资源隔离；
- 本体、GraphRAG、Agent 上下文和治理的一体化能力。

因此，对外竞争话术不宜否认 PostgreSQL 的图能力，而应把它定义为“图查询普及层”，把专用图数据库定义为“关系智能生产层”。

## 五、金融科技与数据治理：Agent 治理需要从统一控制升级为分级自治

Gartner 近期提出，不应对所有 Agent 采用完全相同的治理方式。不同 Agent 在自主程度、影响范围和信任边界上存在显著差异；治理过严会阻碍简单 Agent 的使用，治理过松则会放大高自主 Agent 的风险。其建议是按照 Agent 自主等级配置不同控制要求。

参考：[Gartner Says Applying Uniform Governance Across AI Agents Will Lead to Enterprise AI Agent Failure](https://www.gartner.com/en/newsroom/press-releases/2026-05-26-gartner-says-applying-uniform-governance-across-ai-agents-will-lead-to-enterprise-ai-agent-failure)。

这个观点很适合金融行业。银行内部的 Agent 至少可以分为四类：

1. **只读型 Agent**：查询制度、解释代码、检索知识，不改变系统状态；
2. **建议型 Agent**：生成规则、风险提示或审批建议，由人确认后执行；
3. **受控执行型 Agent**：在明确范围内调用工具和修改业务对象；
4. **高自治型 Agent**：跨系统规划并连续执行多步任务。

不同等级应对应不同治理措施：

- 只读型重点控制数据范围、脱敏和访问日志；
- 建议型还要记录证据、模型版本和人工确认；
- 受控执行型需要动作白名单、额度限制、事务保护和回滚；
- 高自治型需要全过程策略评估、跨系统影响分析和实时中止机制。

图数据库特别适合实现这种分级治理，因为权限不是简单的角色表，而是由用户、Agent、任务、数据、工具、环境和业务对象共同决定。通过图模型，可以动态计算某个 Agent 在特定任务下的有效权限和潜在影响范围。

## 六、公司与市场判断

### 1. 数据库厂商正在争夺 Agent 控制权

EDB 强调数据层策略执行，Oracle 推出数据库内置 A2A 服务，Neo4j 强化图与向量融合，说明数据库市场的竞争已经不再局限于存储引擎。未来数据库产品会越来越多地包含：

- Agent 身份与连接管理；
- 面向 MCP、A2A 等协议的服务接口；
- 数据访问和动作策略；
- 向量、图、全文与关系查询的混合执行；
- Agent 行为日志、审计和成本分析。

数据库厂商的长期目标，是让 Agent 围绕数据平台运行，而不是让数据库成为可替换的数据源。

### 2. 图数据库的最佳位置不是替代关系数据库，而是连接多个执行系统

关系数据库可以在本地强制执行权限，但企业的 Agent 通常跨越 CRM、ERP、数据仓库、文件系统、消息系统和外部服务。任何单一数据库都无法完整理解这种跨系统任务。

图数据库更适合承担全局关系层：连接 Agent、系统、工具、业务对象、数据资产、规则和责任主体，形成一张实时治理图。其价值不是保存每一份原始数据，而是维护跨平台的语义关系和影响路径。

因此，一个更现实的竞争策略是：

> 与关系数据库、湖仓和业务系统协同，成为 Agent 时代的跨系统上下文与治理中枢。

## 七、对工作的可用要点

### 1. 建议把“Agent Control Plane”进一步具体化为两级架构

对外方案中可以明确区分：

- **Data Enforcement Plane**：由数据库和业务系统执行本地权限、行级控制、事务和动作限制；
- **Agent Governance Graph**：由图数据库和本体管理跨系统身份、上下文、策略、依赖、证据和影响。

这样比笼统地讲“统一治理平台”更有说服力，也能解释为什么客户已有 IAM、数据库权限和 API 网关后，仍然需要图数据库。

### 2. 建议新增“上下文图”产品叙事

可以把现有知识图谱、GraphRAG、本体和 Agent 能力统一为“企业上下文图”：

> 企业上下文图持续连接业务对象、数据、人员、规则、流程和 Agent，并记录它们的实时状态变化，为智能问答、主动发现、自动执行和审计治理提供统一上下文。

这个概念比单独强调知识图谱更贴近 Agent 时代，也更容易连接实时数据和行动闭环。

### 3. 建议构建金融 Agent 分级治理 Demo

可以围绕监管报送设计四级 Agent：

- 规则检索 Agent；
- 规则发现与解释 Agent；
- 报送数据预审 Agent；
- 跨系统整改执行 Agent。

演示每一级 Agent 的权限、证据要求、人工确认、动作范围和审计链不同，并通过图查询展示：某个 Agent 为什么有权执行、读取了哪些数据、依据哪些规则、影响哪些报表和业务对象。

这会比单纯展示自然语言问数更接近银行的真实治理需求。

### 4. 向量能力的下一阶段应关注成本和混合优化

竞品已经开始利用量化和二阶段重排优化向量检索。产品规划中应重点评估：

- 向量量化与全精度重排；
- 图过滤与向量检索的代价顺序；
- 大规模 embedding 的内存与存储成本；
- 图路径、规则和向量相似度联合评分；
- 混合查询的执行计划和 Profile 可视化。

这既是性能能力，也是未来 GraphRAG 产品能否进入生产的关键。

### 5. 面对 PostgreSQL，应从“功能对比”转向“工作负载证明”

客户不会因为图数据库支持更多语法就一定采购。最有效的竞争方式是建立可复现的工作负载证明：在五跳以上路径、复杂约束路径、图算法、实时关系更新、影响分析和 Agent 治理查询上，展示专用图数据库在性能、可维护性和可解释性方面的优势。
